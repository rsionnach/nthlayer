## 4. Recommended execution order

Reasoning grounded in dependency edges surfaced by the audits.

1. **`Correct opensrm-zfyh bead text`** (P3, trivial) — prevents the next audit cohort from repeating the same wrong-path mistake (`nthlayer-common-audit-2026-06-05.md:46`). Cheap, do first.
2. **`Update testing.md to match ecosystem reality`** (P2) — the doc is the reference everything else cites. Updating it last would mean re-doing the CI standardisation work twice; updating it first lets every downstream bead reason against accurate prose.
3. **`Cleanup nthlayer-common`** (P2) — it's a transitive dep of every other repo (`nthlayer/CLAUDE.md` §Components). Any test-fixture or shared-builder changes propagate downstream; common-first keeps later beads stable.
4. **`Cleanup nthlayer-core`** (P2) decision-first — its "zero conftests, real-store, zero own-mocking" stance (`nthlayer-core-audit-2026-06-05.md:31, 57, 67-69`) is either the reference architecture or a deliberate outlier. Deciding sets the discipline for the bench/workers/generate cleanups that follow.
5. **`Restore generate test collection`** (P1 sub-bead) — 100 collection errors and 3/3 red CI on main since 2026-05-10 is an active bleeding wound (`nthlayer-generate-audit-2026-06-05.md:24, 38`). Until collection is clean, no further generate work is testable. Can run in parallel with common cleanup if Phase 2 staffing allows.
6. **`Cleanup nthlayer-bench, nthlayer-workers, nthlayer-generate Phase 2 body (post-collection-fix)`** (P2, parallelisable) — all three depend on common landing + testing.md being accurate. Order between the three is staffing-driven, not technical.
7. **`Shared mocking-classification discipline`** (P2) — could lead 5/6/7 if a shared discipline saves total work; could trail if per-repo autonomy is preferred. Surface as ambiguous (single bead vs five per-repo passes).
8. **`Cleanup nthlayer front-door integration scripts`** (P2) — independent of the Python cleanups; can run any time after the testing.md update (so the `test/integration-chain.sh` rewrite-vs-retire decision reasons against current doc).
9. **`CI standardisation across ecosystem`** (P2) — must trail the per-repo cleanups; otherwise CI gates against rotted state and either blocks all future PRs or normalises noise.
10. **`Link testing.md from each repo README/CONTRIBUTING`** (P3) — trail end. Mechanical.
11. **`Pre-commit hook standardisation`** (P3) — optional; defer until evidence supports it.

Out-of-epic but worth tracking in parallel: **`Triage publish-docker.yml red`** — independent track, no test-suite-maintenance gating, but flag so it isn't lost.

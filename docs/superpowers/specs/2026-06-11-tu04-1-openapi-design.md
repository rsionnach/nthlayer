# tu04.1 — Core API reference (OpenAPI 3.1)

**Bead:** `opensrm-tu04.1` (P2)
**Parent:** `opensrm-tu04` (Phase 3-DOC epic)
**Date:** 2026-06-11
**Repo:** all implementation lands in `nthlayer-core/`

## Problem

`nthlayer-core` exposes 29 Starlette routes across 9 path groups:

| Group | Routes | Methods |
|---|---|---|
| `/health` | 1 | GET |
| `/verdicts*` | 7 | POST, GET, GET ancestors/descendants, POST outcome/override |
| `/assessments` | 2 | POST, GET |
| `/cases*` | 6 | POST, GET, GET by id, PUT/DELETE lease, PUT resolve |
| `/change-freezes*` | 3 | POST, GET, PUT lift |
| `/heartbeats` | 1 | POST |
| `/manifests*` | 4 | POST, GET, GET by service, POST reload |
| `/monitoring/*` | 1 | GET stuck-action-requests |
| `/suppressions` | 1 | GET |
| `/component-state/*` | 1 | GET |

There is no OpenAPI spec today. Consumers (workers, bench, external integrators) discover the API from handler source code, ad-hoc docs, and `test/three_tier_assertions.py`. The bead acceptance is strict: an OpenAPI 3.1 spec for **every** endpoint with description, request/response schemas, error codes — and it must lint clean.

## Design

### Architecture

A new module `src/nthlayer_core/openapi_spec.py` holds the spec as a structured Python dict:

```python
OPENAPI: dict = {
    "openapi": "3.1.0",
    "info": {"title": "NthLayer Core API", "version": "1.6.0", "description": "..."},
    "servers": [{"url": "/", "description": "Same-origin"}],
    "paths": {"/health": {...}, "/verdicts": {...}, ...},
    "components": {"schemas": {"ErrorEnvelope": {...}, "Verdict": {...}, ...}},
}
```

The dict is the source of truth. The bead description's phrase "auto-generated from Starlette routes + descriptions" is satisfied by **parity enforcement** rather than runtime introspection: a test asserts that every `(path, method)` in `server.routes` has a matching entry in `OPENAPI["paths"]`, and vice versa. Drift fails CI.

Two artefacts ship from one source:
- **Runtime:** `GET /openapi.json` → `JSONResponse(OPENAPI)`.
- **Static:** `docs/api/openapi.json`, regenerated by `scripts/regen_openapi.py`. CI asserts the checked-in file matches what the dict produces. PRs that change the spec MUST commit the regenerated JSON.

### Why a dict, not introspection

Three reasons:

1. **Existing handlers are dict-based.** They take `Request`, manually parse JSON, return `JSONResponse(record)`. There are no Pydantic models to introspect. Building Pydantic models for all 29 endpoints would be a refactor outside this bead's scope.
2. **The "descriptions" already exist in handler docstrings** — but Starlette's `SchemaGenerator` requires YAML inside docstrings, which is awkward to read, awkward to edit, and unparseable by IDEs. A structured dict separates documentation from code while keeping both in the same repo.
3. **Parity enforcement gives the same guarantee as introspection.** If a route appears in `server.routes` but not in `OPENAPI["paths"]`, the parity test fails — exactly the same drift signal you'd get from runtime introspection.

### Shared schema components

Defined once in `OPENAPI["components"]["schemas"]`, referenced via `$ref` everywhere:

- `ErrorEnvelope` — `{"error": "string", "detail": {"type": "object"}}`. Every 4xx/5xx response.
- `Verdict`, `Assessment`, `Case`, `ChangeFreeze`, `Manifest`, `Heartbeat`, `CaseLease`, `ComponentState` — record shapes. Each documents fields, types, required-ness, sourced from `nthlayer-common/records/` canonical models.
- `EventEnvelope` — CloudEvents v1.0 wrapper that `post_verdict` auto-detects.

### Error-code catalogue

Every error code that handlers emit (`internal_error`, `duplicate`, `verdict_invalid`, `not_found`, `envelope_invalid`, etc.) is documented in the response schemas. The error envelope's `error` field has an `enum` listing all known codes; new codes added to handlers must be added to the enum (parity-tested separately if a registry test materialises in a follow-up — out of scope here).

### Tests

`tests/test_openapi.py` (new):
1. `test_route_parity` — `_route_spec_pairs() == _spec_path_method_pairs()`. Catches any drift.
2. `test_spec_is_valid_openapi_31` — `openapi_spec_validator.validate(OPENAPI)` raises nothing.
3. `test_checked_in_artefact_matches` — `json.load(open("docs/api/openapi.json")) == OPENAPI`.
4. `test_openapi_endpoint_served` — Starlette `TestClient` → `GET /openapi.json` returns 200, JSON body equals `OPENAPI`.

### CI wiring

- `openapi_spec_validator` added to `nthlayer-core`'s `[project.optional-dependencies].dev` (or the equivalent ruff/pytest dev block).
- No new CI step needed — the four tests run as part of `uv run pytest`.
- Spectral or other Node-based linters explicitly **not** adopted (out of scope; openapi-spec-validator is sufficient for the "lint-validated" acceptance).

### Phasing — split into 4 beadlets

Documenting all 29 endpoints with full schemas is ~800 lines of OPENAPI dict, beyond a single-session bead. The work splits into four child beads under `tu04.1`, with `1.1` blocking `1.2`/`1.3`/`1.4`:

| Beadlet | Scope | Routes | Schemas added | Parallel? |
|---|---|---|---|---|
| `tu04.1.1` | Framework + `/health` | 1 | `ErrorEnvelope` | — (blocks others) |
| `tu04.1.2` | `/verdicts*` | 7 | `Verdict`, `EventEnvelope` | with `1.3`, `1.4` |
| `tu04.1.3` | `/assessments` + `/cases*` | 8 | `Assessment`, `Case`, `CaseLease` | with `1.2`, `1.4` |
| `tu04.1.4` | remainder | 10 | `ChangeFreeze`, `Manifest`, `Heartbeat`, `ComponentState` | with `1.2`, `1.3` |

`tu04.1` itself closes when all four beadlets close. Phases 2-4 can fan out as parallel worktree-isolated agents because each touches a disjoint slice of `OPENAPI["paths"]` and `OPENAPI["components"]["schemas"]` — the orchestrator merges their structured returns into the single source-of-truth dict.

## Out of scope

- Swagger UI / Redoc / any HTML rendering of the spec (separate question — could live in `docs-site/` later).
- Pydantic model adoption for request validation (a refactor; not a doc task).
- Spectral or other style-rule linters beyond OpenAPI 3.1 schema validity.
- A separate registry of error codes with its own test (could be a follow-up).
- Versioning the spec independently of the package (`info.version` tracks the package version; no separate spec versioning).

## Test plan (for tu04.1, satisfied collectively by the 4 beadlets)

- `uv run pytest tests/test_openapi.py` exits 0 on every beadlet.
- `python -m openapi_spec_validator docs/api/openapi.json` exits 0 after each beadlet lands.
- After all four close: every `(path, method)` in `server.routes` has a matching `OPENAPI["paths"]` entry — parity test passes.
- `GET /openapi.json` against a running server returns the same JSON as the checked-in `docs/api/openapi.json`.

## Verification → close

`tu04.1` closes when:
- All four `tu04.1.*` beadlets close.
- `docs/api/openapi.json` exists, lints clean, covers all 29 routes.
- Test suite green.
